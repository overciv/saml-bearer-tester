Step 2 — DPoP Key Pair
DPoP keys are ephemeral by design. Generate a new key pair for each testing session (or client). The public key is embedded directly in every DPoP proof JWT — Okta doesn't need to store it. The access token will contain cnf.jkt = SHA-256 thumbprint of this public key.

Step 4 — Resource Access
A new DPoP proof must be generated for each resource request. It includes ath = BASE64URL(SHA-256(access_token)), binding the proof to the specific token being used. The resource server validates this binding before accepting the request.

Resource server validation steps:
  1. Decode the DPoP proof JWT from the DPoP header
  2. Extract the jwk public key from the proof header
  3. Verify the proof signature using that public key
  4. Check htm and htu match the actual request
  5. Verify iat is recent (within 300s) and jti is unique
  6. Calculate JKT = SHA-256 thumbprint of the embedded JWK
  7. Extract cnf.jkt from the access token (via introspect or local decode)
  8. Assert JKT from proof == cnf.jkt from token
  9. Verify ath = BASE64URL(SHA-256(access_token))
